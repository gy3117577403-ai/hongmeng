# 备份计划 v1.5.0

## 备份目标
系统数据由两部分组成：
- PostgreSQL：保存用户、工单、资源分类、文件元数据和操作日志。
- Object Storage：保存 PDF、JPG、PNG 等实际文件。

完整恢复必须同时保证 PostgreSQL 元数据和对象存储文件可用。

## PostgreSQL 备份策略
- 使用 Sealos PostgreSQL 的备份能力。
- 建议每日自动备份。
- 重要上线前创建手动快照。
- 备份应保留足够周期，便于追溯误操作。
- 定期检查备份任务是否成功。

## Object Storage 备份策略
- 生产 Bucket 长期保留。
- 对重要资料做额外归档，可使用跨 Bucket、跨区域或离线方式。
- 不要自动清理未知对象。
- 文件软删除只影响业务展示，不应物理删除对象存储中的历史文件。

## 建议频率
- 数据库：每日备份。
- 对象存储：长期保留，按业务重要性做周期归档。
- 重要上线前：手动快照数据库，并确认对象存储状态正常。

## 恢复演练建议
至少每个重要版本周期执行一次恢复演练：
1. 准备非生产环境。
2. 恢复 PostgreSQL 备份。
3. 连接测试用 Object Storage Bucket。
4. 启动应用。
5. 验证登录、工单列表、文件预览和下载。
6. 记录恢复耗时和发现的问题。

恢复演练不要直接在生产环境执行。

## 数据不丢失验收步骤
1. 登录系统。
2. 新建测试工单。
3. 上传 PDF 文件。
4. 上传图片文件。
5. 刷新页面，确认文件仍存在。
6. 退出后重新登录，确认工单和文件仍存在。
7. 重启或重新部署应用容器，确认数据仍存在。
8. 检查 PostgreSQL 中存在对应文件元数据。
9. 检查 Object Storage 中存在对应实际文件。
10. 软删除文件后，确认前端默认隐藏，数据库记录仍保留。

## 备份责任建议
- 上线前：发布负责人确认备份或快照。
- 日常：运维负责人检查自动备份状态。
- 故障时：先保留现场日志和截图，再执行恢复。

## 注意事项
- 不要把数据库连接串、对象存储密钥或任何密码写入备份记录。
- 不要删除生产 Bucket。
- 不要在没有备份的情况下执行破坏性操作。

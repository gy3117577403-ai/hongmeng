# 生产闭环数据审计与验收说明

日期：2026-07-24

范围：生产工序完成、数量流转、不良品分支、工时池及员工领取

模式：只读，不修改工单、路线、状态、数量、工时或权限

## 本阶段交付

新增统一的生产闭环审计入口：

```powershell
npm run audit:production-closure
```

审计在 PostgreSQL `REPEATABLE READ` 快照中执行，并在事务开始后设置
`READ ONLY`。因此同一次报告读取到的数据口径一致，且数据库会拒绝事务内
的写操作。

可选参数：

```powershell
npm run audit:production-closure -- --json
npm run audit:production-closure -- --output .\tmp\production-closure-audit.json
npm run audit:production-closure -- --fail-on-warning
npm run audit:production-closure -- --no-fail
```

- 默认发现错误时返回非零退出码，适合 CI 或上线前检查。
- `--fail-on-warning` 会把警告也作为失败处理。
- `--no-fail` 只输出报告，不因数据问题改变退出码。
- 报告不输出数据库地址、密码、Token 或对象存储密钥。

建议目标数据库账号本身也只授予查询权限，形成应用层和数据库层的双重只读
保护。生产环境执行仍需单独授权，本阶段未连接或操作 Sealos。

## 审计规则

### 工单与工艺路线

- 累计完成数量不得超过生产目标。
- 数量关系满足：完成数量 ≤ 已转序数量 ≤ 生产目标。
- 工单标记完成时，工艺路线、目标数量、完成时间和所有分支必须同时闭环。
- 路线标记完成时，全部工序必须为已完成或已跳过，并存在完成时间。

### 工序与数量移动

- 工序数量满足：已处理 = 良品 + 不良品。
- 已处理不得超过投入，已释放不得超过良品。
- 工序累计值必须等于有效完成记录与返工回流记录的合计。
- 首组工序投入必须等于生产目标，后续工序投入必须等于有效入站移动合计。
- 并行工序组使用共同释放量，且不能超过组内最小良品产出。
- 累计释放量必须与转序、成品或返工回流移动记录一致。

### 不良品与分支工单

- 有不良品必须选择返工、报废补产或质量待判，并建立唯一分支。
- 分支类型、目标数量、来源工序、父工单、根工单和拆分移动必须一致。
- 非返工分支不能配置回流工序。
- 分支标记已解决时，其路线和完成数量必须闭环。

### 工时池与员工领取

- 工时池满足：可领取数量 = 已领取数量 + 剩余数量。
- 标准工时满足：总标准工时 = 已领取标准工时 + 剩余标准工时。
- 工时池状态必须与领取进度一致。
- 工时总量按照完成时冻结的计时方式、单位工时、准备工时和单位次数重新核算。
- 有效员工领取记录合计必须等于工时池累计领取值。
- 冲销记录必须存在等额、反向、同员工、同工时池的反向记录。
- `800 + 200` 分拆领取必须精确领完同一个 `1000` 数量工时池，不产生舍入丢失。

## 验收边界

本阶段验证的是数据模型、事务结果和跨表守恒关系，不改变现有接口或业务逻辑。
它不能代替现场人员对实际工艺路线、标准工时数值和权限分工的业务确认。

进入生产数据验收前，应先：

1. 使用仅查询账号连接目标数据库；
2. 在低峰期执行审计并保存 JSON；
3. 按错误代码逐项确认历史数据与当前业务规则的差异；
4. 未经确认不自动修复历史记录；
5. 审计无错误后，再进行主管端与员工端的现场 UAT。

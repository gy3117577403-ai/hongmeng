# 回滚指南 v1.5.0

## 什么时候需要回滚
出现以下情况时可以考虑回滚：
- 新版本启动失败，且短时间无法修复。
- 核心流程不可用，例如登录、工单列表、上传、预览或下载失败。
- 新版本出现严重性能问题，影响生产现场使用。
- 应用日志持续出现阻塞性错误。

如果只是非阻塞 UI 文案或轻微体验问题，优先记录问题并评估是否补丁修复。

## 回滚前准备
回滚前必须保留现场信息：
1. 截图保存 Sealos 应用状态。
2. 保存应用最近日志。
3. 记录当前镜像地址和 digest。
4. 记录故障发生时间、操作步骤和影响范围。
5. 确认数据库和对象存储没有被删除或重建。

## 如何在 Sealos 里回滚镜像
1. 打开 Sealos 应用配置。
2. 找到镜像地址。
3. 将镜像改回上一稳定版本的完整 digest 地址。
4. 保存并重新部署应用。
5. 等待容器启动完成。

建议使用完整 digest 镜像地址，避免 `latest` 指向变化带来的不确定性。

## 回滚后验证
回滚完成后执行以下检查：
- `/api/health` 返回正常。
- 登录可用。
- 工单列表可打开。
- 上传 PDF 或图片可用。
- 文件预览可用。
- 单文件下载可用。
- 当前工单全部资料 ZIP 下载可用。
- 操作日志可查看。

## 数据库 migration 已执行后的注意事项
v1.5.0 之前已经通过 Prisma migration 管理数据库结构。回滚代码时注意：
- 不要手动删除 migration 表。
- 不要回滚数据库结构，除非已经有明确、可验证的数据库回滚方案。
- 若新版本 migration 只新增字段，旧代码通常可以继续读取已有数据。
- 若未来版本包含破坏性 migration，必须先制定数据库回滚方案再上线。

## 禁止操作
- 不要删除数据库。
- 不要删除对象存储 Bucket。
- 不要清空对象存储文件。
- 不要在不确认备份可用的情况下重建生产环境。
- 不要把任何密钥或密码写入回滚记录。

## 回滚完成记录
回滚后建议记录：
- 回滚时间
- 回滚前镜像 digest
- 回滚后镜像 digest
- 验证人
- 验证结果
- 后续修复计划

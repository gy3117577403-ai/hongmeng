# 生产闭环本地发布门禁与角色验收

日期：2026-07-24

范围：生产调度完成并转序、工时池、员工领取、出勤达成率和角色权限

边界：本门禁不连接目标数据库，不访问 Sealos，不写入生产工单或员工工时。

## 一键本地门禁

```powershell
npm run verify:production-closure
```

该命令依次执行：

1. 生产闭环聚焦测试；
2. Next.js 生产构建、TypeScript 检查和现有 ESLint 检查。

聚焦测试也可以单独执行：

```powershell
npm run test:production-closure
```

聚焦范围包括：

- 考勤有效工时与 95% 达成率基数；
- 员工、班组长、管理员的工时领取权限；
- 完成数量、良品、不良品和处置方式守恒；
- 顺序工序、并行工序与返工分支；
- `800 + 200` 分拆领取和标准工时守恒；
- 工单、路线、工序、工时池和领取记录审计；
- 生产写操作和员工自助操作的请求权限边界。

数据库集成测试不属于本地门禁默认步骤，只有准备好隔离测试数据库时才单独执行：

```powershell
$env:RUN_DB_INTEGRATION = "1"
npm run test:process-integration
```

## 生产主管验收路径

使用具备生产操作权限的主管账号，在隔离验收环境检查：

1. 打开 `/production`，订单主操作显示“下一步”或“完成当前工序 → 下一道工序”，不出现生产总量校正入口。
2. 打开下一步弹窗，当前工序和下一工序与已发布产品工艺路线完全一致。
3. 输入本次处理数量和不良品数量，确认良品数量自动计算且三者满足“处理数 = 良品 + 不良品”。
4. 有不良品时必须选择返工或报废补产；未选择时提交被阻止。
5. 选择现场作业人员仅作为领取推荐，不直接把工时计入员工。
6. 提交后刷新调度中心，良品只进入路线规定的下一工序；不良品形成独立分支。
7. 打开 `/workspace/reports?view=labor`，确认该完成批次形成当日工时池。
8. 主管只能为权限范围内的员工分配工时；管理员可处理全量和待补标准工时。

验收过程中使用专用测试工单。不得用真实生产工单试点，除非另行明确授权。

## 员工验收路径

使用已绑定员工档案的员工账号，在隔离验收环境检查：

1. 打开 `/workspace/reports?view=labor`，只能选择本人，不能切换为其他员工。
2. 领取部分数量后，工时池的已领取和剩余数量同步变化。
3. 另一员工领取剩余数量后，工时池显示已领完；累计领取数量等于可领取数量。
4. 员工不能冲销领取记录、补录工时标准或执行生产转序。
5. 回到“员工达成率”，领取记录归属工序完成日期；当日缺少确认考勤时显示待匹配。
6. 确认考勤后，达成率按“标准完成工时 ÷（有效出勤 × 95%）”计算。

## 验收通过条件

- `npm run verify:production-closure` 返回退出码 0；
- 主管和员工的页面入口、按钮范围与角色权限一致；
- 工序只按已发布路线推进；
- 不良品全部进入可追溯分支；
- 工时只在工时池领取后计入员工；
- 分拆领取无数量或毫秒级标准工时丢失；
- 本阶段不产生目标数据库或 Sealos 变更。

## 当前结论

本地门禁是提交和部署前的固定检查，不替代目标数据库审计或现场真实账号 UAT。
本轮按项目安排跳过数据库与 Sealos 阶段，后续如需恢复，应单独授权并使用隔离
工单或只读账号。
